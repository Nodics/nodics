# Licensing And Header Contract

This page defines the Nodics file-header and licensing rule for framework
source, generated files, templates, examples, documentation, and customer
project skeletons.

The rule is intentionally simple: the legal/header contract is part of the
platform governance surface. Humans, AI tools, generators, and fix commands
must apply the same standard instead of inventing file-by-file wording.

## Standard Source Header

Every JavaScript file that is owned by Nodics framework source, generated by
Nodics tooling, or created as a Nodics project/module/server/node skeleton must
start with this header:

```javascript
/*
    Nodics - Enterprice Micro-Services Management Framework

    Copyright (c) 2026 Nodics All rights reserved.

    This software is the confidential and proprietary information of Nodics ("Confidential Information").
    You shall not disclose such Confidential Information and shall use it only in accordance with the
    terms of the license agreement you entered into with Nodics.

 */
```

Files with a Unix shebang, such as executable Node.js scripts, keep the shebang
as the first line and place the standard header immediately after it.

## Where The Header Is Required

The standard header is required for:

- framework runtime source files;
- configuration, lifecycle, router, schema, service, facade, controller,
  pipeline, interceptor, utility, data, and test JavaScript files;
- generated JavaScript files under generated controller, facade, service,
  schema, API, CRUD, test, and context surfaces;
- module, group, project, environment, server, and node skeleton JavaScript
  files created by structure generation;
- JavaScript examples that are intended to be copied into Nodics source or
  project modules.

Generated artifacts are not hand-maintained. The generator must emit the header
from the canonical tooling contract, and `npm run quality:copyright:fix` may
normalize generated output when the canonical text changes.

## Documentation And Markdown

Markdown documentation does not need the JavaScript block header at the top of
every file. Public documentation, README files, AGENTS files, and LLM contracts
should explain behavior, ownership, extension paths, and governance rules
without adding source-style boilerplate to every page.

When a documentation page shows a JavaScript skeleton or template, the snippet
must use the current 2026 standard source header.

## Third-Party And Excluded Files

Do not overwrite third-party compatibility code or vendor-owned files with the
Nodics header unless Nodics owns the file. Exclusions must be explicit in the
copyright quality service and documented by reason.

The current governance excludes `gFramework/nConfig/bin/enum.js` because it is
a bundled third-party enum compatibility implementation.

## Customer Project Skeletons

Generated customer project skeletons follow the same header unless a customer
project has a legally approved replacement contract. A project-specific legal
header must be implemented by replacing or extending the generator and copyright
quality service through governed project tooling.

Do not change header text inside individual generated files after generation.
Change the canonical generator/checker and regenerate or normalize the output.

## Enforcement

Use:

```bash
npm run quality:copyright
npm run quality:copyright:fix
```

The enforced checker is owned by nTooling. It validates all JavaScript files
that are in repository scope, skips private reference material under `docs/`,
and preserves shebangs while normalizing the standard header.

Before committing a header or licensing change, run:

```bash
npm run quality:copyright
npm run test:tooling
npm run llm:validate
```

Run broader gates when the change also affects generators, generated artifacts,
module structure, or release contracts.
